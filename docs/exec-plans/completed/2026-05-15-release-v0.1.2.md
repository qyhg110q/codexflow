# Goal

Publish CodexFlow `v0.1.2` to GitHub Releases with rebuilt Windows host, Web, and Android assets, while aligning app-visible version metadata and writing release notes from the commits after `v0.1.1`.

# Scope

- Bump release-facing version strings needed by the Flutter client and Go Agent.
- Generate `v0.1.2` release assets from a clean `HEAD` snapshot.
- Write release notes based on the Git history from `v0.1.1` to the latest commit.
- Create the `v0.1.2` tag and publish a GitHub Release with packaged assets.

Out of scope:

- New product features beyond what already landed after `v0.1.1`.
- iOS signed distribution.
- Cleaning unrelated generated files or local runtime artifacts in the main worktree.

# Surprises & Discoveries

- The source tree at `HEAD` still reported `0.1.0+1`, so a release without version alignment would have produced a mismatched APK and fallback app version.
- The main worktree contains generated Flutter plugin files plus local helper scripts and logs that are unrelated to the release target.
- GitHub publishing does not require `gh` on this machine because the repo already includes a REST-based publishing script with credential-manager fallback.

# Decision Log

- Build from a clean temporary worktree rooted at the latest commit instead of the dirty main worktree.
- Keep the release scope limited to version alignment, packaging, validation, and publication. Do not absorb unrelated local changes.
- Use commit-derived release notes rather than the generic boilerplate emitted by `build_release_assets.ps1`.
